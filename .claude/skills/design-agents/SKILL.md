---
name: design-agents
description: 指示された開発内容(例「ECサイトを開発するエージェント設計をして」)に応じて、projects/ 配下にプロジェクト専用の Claude Code エージェント環境(CLAUDE.md・スキル・サブエージェント・settings.json)を設計・構築する。「〜を開発するエージェントを作って」「〜の開発体制を整えて」という依頼で使う。
---

ユーザーの指示: $ARGUMENTS

以下のフェーズを順に実行する。references/ と templates/ はこのリポジトリのルート
(CLAUDE.md があるディレクトリ)からの相対パス。

## フェーズ1: 要件分析

1. `requirements-analyst` サブエージェントに、ユーザーの指示文をそのまま渡して
   要件分析させる(対象ドメイン、技術スタック、規模、リスク、繰り返し作業の候補、不明点)。
   指示文が要件定義フォーム(tools/requirements-form.html)の出力形式(【目的】【体制の種類】…)
   の場合もそのまま渡す。フォーム由来の回答は確定事項として扱い、質問を最小限にする。
2. 返ってきた不明点のうち、**設計が分岐する重大なもの**(技術スタック未定、規模感、
   品質要求レベルなど)が残る場合のみ、AskUserQuestion でユーザーに確認する。
   推測で埋められるものは推測し、その旨を最終報告に明記する。質問は最大1回にまとめる。

## フェーズ2: アーキテクチャ設計

3. `architecture-designer` サブエージェントに、確定した要件を渡して設計書を作らせる。
   設計書には次を含めさせる:
   - プロジェクト名(英語ケバブケース)と生成先 `projects/<名前>/`
   - 採用パターンとその根拠(references/design-patterns.md の用語で)
   - 生成ファイル一覧(各ファイルの目的を1行ずつ)
   - 各スキル・サブエージェントの名前、責務、入出力形式
   - 体制提示書(design-doc.html)の章立て(どのセクション部品を使うか)
4. 設計書の要約(採用パターン、ファイル一覧、判断根拠)をユーザーに提示してから構築に進む。
   ユーザーの承認待ちはしない(疑問があればフェーズ1で解消済みのはず)。

## フェーズ3: 構築

5. 設計書に従い、`projects/<名前>/` 配下にファイルを生成する。
   - 形式は references/claude-code-formats.md に厳密に従う。
   - 雛形として templates/ 配下のテンプレートを下敷きにする。
   - 生成する CLAUDE.md・スキル本文・README は日本語で書く。
   - 必ず生成するもの: CLAUDE.md、README.md、.claude/settings.json、
     docs/design-doc.html(体制提示書)。
   - スキル・サブエージェントは設計書にあるものだけを生成する(勝手に追加しない)。

### 体制提示書(docs/design-doc.html)の生成ルール

- templates/design-doc.template.html を下敷きにする。これは固定フォームではなく
  部品カタログである(テンプレート冒頭のコメントに従う)。
- 「概要」「体制(誰がいるか)」は必須。それ以外の章は設計の種類に合わせて
  取捨選択する(開発体制なら分担+ワークフロー、調査系なら成果物の流れ、
  運用自動化なら定期実行スケジュール+エスカレーション基準、など)。
  テンプレート末尾の部品カタログから必要なものを組み込む。
- {{...}} プレースホルダを残さない。生成日はその日の日付を入れる。
- 内容は実際に生成した CLAUDE.md・スキル・エージェントと完全に一致させる。
  提示書だけに存在する役割や、提示書に載っていないスキルを作らない。
- 外部リソース(CDN等)を参照しない自己完結HTMLを維持する。

## フェーズ4: レビューと報告

6. `design-reviewer` サブエージェントに `projects/<名前>/` のパスを渡してレビューさせる。
7. 指摘のうち妥当なものを修正する(形式違反・誘導漏れ・矛盾は必ず直す。好みレベルの
   指摘は見送ってよい)。
8. ユーザーへの最終報告に必ず含めること:
   - 生成先パスとファイル一覧
   - 採用した設計パターンと理由
   - 推測で埋めた前提(あれば)
   - 体制提示書のパス(projects/<名前>/docs/design-doc.html をブラウザで開けば
     体制の全体像を確認できること)
   - 使い始め方(既存リポジトリへのコピー手順、最初に試すスキル)

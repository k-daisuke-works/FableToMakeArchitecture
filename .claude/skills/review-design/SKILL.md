---
name: review-design
description: projects/ 配下に生成済みのエージェント環境を監査し、形式違反・使われない構成物・誘導漏れを見つけて改善する。「この設計を見直して」「〜プロジェクトの体制を改善して」という依頼で使う。
---

対象: $ARGUMENTS(未指定なら projects/ 配下を一覧してユーザーに対象を確認する)

1. `design-reviewer` サブエージェントに対象プロジェクトのパスを渡し、監査させる。
2. ユーザーから具体的な不満(「このスキルが動かない」「レビューが機能していない」等)が
   伝えられている場合は、それをレビュー観点としてサブエージェントに追記して渡す。
3. 指摘を重大度順に整理してユーザーに提示する:
   - **要修正**: 形式違反(frontmatter 不備等)、CLAUDE.md からの誘導漏れ、矛盾した指示
   - **推奨**: 過剰な構成物の削除、description の改善、permissions の不足
4. 要修正はそのまま直す。推奨は内容を提示し、ユーザーの指示があれば適用する。
5. 修正後、何をなぜ変えたかを報告する。

## 監査でよくある問題(design-reviewer に渡す標準観点)

- スキルの description が曖昧で自動起動しない
- サブエージェント本文に最終出力の形式指定がない
- CLAUDE.md が肥大化している(手順書はスキルへ移すべき)
- 作られたが使い道のないスキル・エージェントが放置されている
- settings.json の deny に秘密情報(.env 等)が入っていない

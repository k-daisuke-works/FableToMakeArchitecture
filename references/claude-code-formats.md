# Claude Code 構成ファイル形式リファレンス

プロジェクト環境を生成するとき、各ファイルはこの形式に厳密に従うこと。

## 1. CLAUDE.md(プロジェクトメモリ)

- 置き場所: プロジェクトルート直下 `CLAUDE.md`
- セッション開始時に毎回コンテキストへ読み込まれる。**簡潔さが命**。
  長文の手順書はスキルに逃がし、CLAUDE.md には「常に守るべきこと」だけを書く。
- 書くべき内容:
  - プロジェクトの一言説明と技術スタック
  - ビルド・テスト・リントの実行コマンド
  - コーディング規約のうち、コードから読み取れないもの
  - 「〜のときは /スキル名 を使う」というスキルへの誘導
  - やってはいけないこと(触ってはいけないファイル、禁止コマンド等)
- 書いてはいけない内容: コードを読めば分かること、一度きりの情報、長い手順書

## 2. スキル(`.claude/skills/<name>/SKILL.md`)

```markdown
---
name: skill-name
description: このスキルを使うべき状況の説明。ユーザーの言葉に近い表現で書くと自動起動の精度が上がる。
---

(本文: Claude への手順指示。番号付きステップで書く)
```

- `name` は英語ケバブケース。ディレクトリ名と一致させる。
- ユーザーが `/skill-name` と打つか、description が状況に合致すると起動される。
- 補助ファイル(チェックリスト、テンプレート等)は同じディレクトリに置き、
  本文から相対パスで参照する。
- 引数を受ける場合は本文中で `$ARGUMENTS` を参照できる。
- 向いているもの: 繰り返し実行する定型ワークフロー(リリース手順、レビュー手順、
  マイグレーション作成、デプロイ確認など)。

## 3. サブエージェント(`.claude/agents/<name>.md`)

```markdown
---
name: agent-name
description: どういうタスクのときにこのエージェントへ委譲すべきかの説明。メインエージェントがこれを読んで委譲を判断する。
tools: Read, Grep, Glob, Bash
model: sonnet
---

(本文: このエージェントのシステムプロンプト。役割、手順、出力形式を書く)
```

- `tools` は省略可(省略時は全ツール継承)。読み取り専用の役割なら
  `Read, Grep, Glob` などに絞ると安全性が上がる。
- `model` は省略可(省略時は継承)だが、役割が明確なら
  references/design-patterns.md「モデル配分とトークン効率」の表に従って指定する
  (定型・高頻度=haiku、調査・通常実装=sonnet、品質ゲート=opus)。迷ったら省略。
- 本文の末尾に「最終出力の形式」を必ず指定する。サブエージェントの最終メッセージ
  だけが親に返るため、形式が曖昧だと親が結果を使えない。
- 向いているもの: コンテキストを汚したくない調査、独立した視点が必要なレビュー、
  並列化したい定型作業。

## 4. settings.json(`.claude/settings.json`)

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run test:*)",
      "Bash(npm run lint)"
    ],
    "deny": [
      "Read(.env)",
      "Bash(rm -rf:*)"
    ]
  }
}
```

- `allow` には、そのプロジェクトで頻繁に使う安全なコマンドを入れて許可プロンプトを減らす。
- `deny` には、秘密情報ファイルの読み取りと破壊的コマンドを入れる。
- チーム共有するものは `settings.json`、個人用は `settings.local.json`(gitignore対象)。

## 5. フック(settings.json 内 `hooks`)

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "npm run lint --silent" }
        ]
      }
    ]
  }
}
```

- 主なイベント: `PreToolUse`(ツール実行前、ブロック可)、`PostToolUse`(実行後)、
  `Stop`(応答完了時)、`UserPromptSubmit`(プロンプト送信時)。
- 「編集のたびに必ずフォーマッタを通す」のような**確実に毎回実行したい**ものだけフックにする。
  Claude の判断に任せてよいものは CLAUDE.md やスキルに書く。
- フックのコマンドは失敗時の挙動(exit code)まで考えて書く。exit 2 で PreToolUse は
  ツール実行をブロックできる。

## 6. 生成するプロジェクトの標準レイアウト

```
projects/<name>/
  CLAUDE.md
  README.md              ← 環境の使い方(人間向け): スキル一覧、想定ワークフロー
  tools/
    request-form.html    ← 要件定義フォーム(人間向け): 項目を埋めるとスキルへの指示文を生成
  .claude/
    settings.json
    skills/<skill>/SKILL.md
    agents/<agent>.md
```

### 要件定義フォーム(tools/request-form.html)

- templates/request-form.template.html を下敷きにした自己完結HTML(外部リソース禁止)。
- メタエージェントの tools/requirements-form.html のプロジェクト版: 人間がブラウザで
  項目を埋めると、そのプロジェクトのスキルにそのまま貼り付けられる指示文
  (/スキル名 + 【ラベル】形式の要件定義書)を生成し、コピー/ファイル保存できる。
- 固定フォームではなく部品カタログ方式: 質問項目は主要スキルが必要とする入力に
  合わせて設計する。複数スキルがあれば「依頼の種類」で切り替える。
- フォームの項目・ラベルは実際のスキルの入力仕様と完全に一致させる。スキルの入力を
  変えたらフォームも更新する。対応するスキル本文にはフォーム出力形式も受け付ける旨を書く。

実際のアプリコードはユーザーが後からこの中で開発する(または既存リポジトリに
`.claude/` 一式をコピーして使う)。README.md にコピー手順も書いておくこと。

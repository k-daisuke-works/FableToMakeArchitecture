# エージェント体制の設計パターン

指示内容からどの体制を選ぶかの判断基準。**デフォルトは最小構成**。
サブエージェントやスキルは「あると便利そう」ではなく「ないと困る理由」があるものだけ入れる。
使われない構成物はノイズになり、CLAUDE.md の誘導も薄まって全体の品質を下げる。

## 規模の目安

| 規模 | 例 | 標準構成 |
|---|---|---|
| S: 小規模・個人 | CLIツール、スクリプト集、静的サイト | パターン1 |
| M: 中規模・継続開発 | Webアプリ、API、モバイルアプリ | パターン1+2(必要なら3) |
| L: 大規模・高リスク | 決済系、マルチモジュール、チーム開発 | パターン2+3+4 |

## パターン1: ソロ+スキル型(デフォルト)

サブエージェントなし。CLAUDE.md + 定型ワークフローのスキル数個。

- スキル候補: 機能開発手順(`/new-feature`)、テスト実行と修正(`/verify`)、
  リリース準備(`/release-prep`)など、**そのプロジェクトで実際に繰り返す作業**。
- 選ぶ条件: 1人開発、コードベースが1コンテキストに収まる、リスクが低い。
- スキルは 2〜4 個まで。使うか分からないものは作らない。

## パターン2: レビューゲート型

実装はメインエージェントが行い、独立した視点のレビュー用サブエージェントを置く。

- サブエージェント候補:
  - `code-reviewer`(読み取り専用: Read, Grep, Glob, Bash)— 差分のバグ・規約違反を指摘
  - `test-runner` — テスト実行と失敗解析に特化
- CLAUDE.md に「機能実装が終わったら code-reviewer に差分レビューを依頼する」と誘導を書く。
- 選ぶ条件: バグの影響が大きい、品質基準が明文化されている、継続的に開発する。
- ポイント: レビュアーは**実装の会話コンテキストを持たない**ことに価値がある。
  差分とレビュー観点だけを渡す設計にする。

## パターン3: 調査分離型

コードベースが大きく、調査でコンテキストが溢れる場合に、調査専用サブエージェントを置く。

- サブエージェント候補:
  - `codebase-explorer`(読み取り専用)— 「Xはどこで実装されているか」に位置と要約で答える
  - `docs-researcher`(WebSearch/WebFetch 付き)— 外部ライブラリ・API仕様の調査
- 選ぶ条件: モノレポ、レガシー大規模コード、外部API依存が多い。
- ポイント: 出力形式を「ファイルパス:行番号 + 3行以内の要約」のように厳しく縛る。

## パターン4: パイプライン型

計画→実装→検証を役割分担する。**本当に必要な場面は少ない**ので安易に選ばない。

- サブエージェント候補: `planner`(読み取り専用・計画立案)、`implementer`、`verifier`。
- 選ぶ条件: タスクが大きく定型的で、同じ流れを何度も回す(例: 大量の類似モジュール移植、
  多数の画面の量産)。
- ポイント: 各段の受け渡し形式(計画書の書式、完了報告の書式)をスキルで固定する。

## 横断的な構成要素(パターンに関わらず検討)

- **verify 系スキル**: 「変更が実際に動くことを確認する手順」(起動コマンド、確認URL、
  代表的な動作確認シナリオ)。ほぼ全プロジェクトで価値がある。
- **settings.json の permissions**: そのスタックの定番コマンド(`npm run test:*` 等)を
  allow に、`.env` 読み取りや破壊的コマンドを deny に。全プロジェクトで必ず生成する。
- **フック**: フォーマッタ/リンタの強制くらいに留める。凝ったフックは壊れやすい。
- **TDD ルール**: テスト文化が求められる場合、CLAUDE.md に「先にテストを書く/
  テストが通るまで完了としない」を明記する(スキルより規約として書く方が効く)。

## モデル配分とトークン効率(全パターン共通のデフォルト)

サブエージェントを置くときは、**役割の誤りコストと実行頻度に応じて model を明示する**。
「各モデルが最効率かつ最大限力を発揮できる構成」を全プロジェクトで標準とする。

| model | 割り当てる役割 | 例 |
|---|---|---|
| haiku | 定型・機械的で判断力が不要、かつ高頻度に回る作業 | テスト実行と結果要約、リント結果整理、ログ集計 |
| sonnet | 調査・要約・通常実装など中程度の判断 | 外部ライブラリ調査、コードベース探索、競合調査 |
| opus | 誤りの損害がトークン単価を上回る品質ゲート | 独立コードレビュー、公開前の内容検証、アーキテクチャ監査 |

- **最大の効き所は高頻度の定型作業**(テスト実行など)。ここを haiku にする効果が最も大きい。
- 迷ったら省略(継承)でよいが、上記に明確に当てはまる役割は必ず指定する。
- モデル配分とセットで、**コンテキスト節約設計**を必ず行う:
  - サブエージェントの最終出力の形式を短く縛る(行数上限、「生ログを貼らない」等)。
  - レビュー系には差分と観点だけを渡し、実装時の会話コンテキストを渡さない。
  - 長大な出力(テストログ・検索結果・ページ本文)はメイン会話に持ち込まず、
    サブエージェント内で消化して要約だけ返させる。
- メインエージェント自体のモデルはユーザーの起動設定に委ね、設計では規定しない。

## 技術スタック別の定石

- **Node/TypeScript**: allow に `npm run *`(または pnpm/yarn)、フックで
  `prettier`/`eslint --fix`。verify スキルに `npm run dev` + 確認手順。
- **Python**: allow に `pytest`、`ruff check`。フックで `ruff format`。
- **Go**: allow に `go test ./...`、`go vet`。フックで `gofmt -w`。
- **不明・混在**: 生成時にスタックを決めつけない。CLAUDE.md にコマンドの
  プレースホルダを置き、README に「最初に埋めよ」と書く。

## 設計時のチェックリスト

1. この構成物(スキル/エージェント)は、週1回以上使われる見込みがあるか?
2. サブエージェントに渡す入力と、返させる出力の形式を定義したか?
3. CLAUDE.md から各スキル/エージェントへの誘導(いつ使うか)を書いたか?
4. permissions の deny に秘密情報と破壊的操作を入れたか?
5. ユーザーがこの環境を使い始める最初の一歩が README に書いてあるか?
6. 各サブエージェントに model(haiku/sonnet/opus)を役割の誤りコスト・頻度に応じて
   指定し、出力形式を短く縛った(トークン効率)か?

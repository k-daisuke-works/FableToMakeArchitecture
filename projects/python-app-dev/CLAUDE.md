Python アプリ開発体制。人間の確認は3つの節目のみ。

## 開発ルール

- **完了の定義**: テストが全て通り、code-reviewer の「要修正」指摘が解消されるまで
  タスクを完了としない。
- **テスト先行**: 実装より先に失敗するテストを書く(pytest)。
- **人間確認の節目は3つだけ**: ①競合調査結果の提示後(作る/既存を使うの判断)
  ②実装計画の確定時 ③完成報告時。それ以外は自律で進める
  (競合調査を省略した機能追加では②③の2つ)。
- 上記の TODO(アプリ種類・フレームワーク・起動コマンド)は、最初の開発の
  計画確定時(節目②)に確定させ、このファイルを更新する。

## スキルとエージェントの使い分け

依頼は自由文のほか、要件定義フォーム(`tools/request-form.html` をブラウザで開いて
生成した【依頼種別】…形式のテキスト/保存した .md)でも受け付ける。

- 新しいアプリを作る・機能や画面を追加する → `/develop`
- 動作確認(整形・リント・テスト・起動)→ `/verify`
- テストの実行は必ず `test-runner` エージェントに委譲する(生ログをメイン会話に
  貼らない。トークン効率のため)。
- 実装完了後は必ず `code-reviewer` エージェントに差分を渡す(実装時の会話は渡さない)。
- 実装前の競合・ライブラリ調査は `competitor-researcher` エージェントに委譲する。

## やってはいけないこと

- .env・秘密鍵・secrets/ 配下を読まない
- `git push --force`・`git reset --hard`・`rm -rf` を実行しない
- テストを削除・スキップして「通った」ことにしない

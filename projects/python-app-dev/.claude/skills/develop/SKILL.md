対象: $ARGUMENTS

`tools/request-form.html` が生成した要件定義書(【依頼種別】から始まる【ラベル】形式の
テキスト。種別により項目は異なる。保存した .md ファイルのパスでも可)も同じように受け付ける。
要件定義書の「未回答(〜してよい)」はユーザーが推測を許可した回答として扱い、
確認せず進める。

1. 入力から要件を整理し、依頼種別(新規アプリ開発/機能追加・画面追加)を判定する。
   競合調査の要否を決める: 新規開発は必須。機能追加は新しいライブラリの導入を伴うとき、
   または【競合調査の要否】が「必要」のときのみ行う(「不要」なら手順4へ)。
2. `competitor-researcher` エージェントに、作りたいものの1〜3行要約・比較観点
   (ライセンス/保守状況/要件カバー度)・ユーザーが特に知りたいことを渡して調査させる。
3. 調査結果と推奨(自作する/既存を使う/組み合わせる)を提示し、
   **続行判断を待つ【節目①】**。ここは人間の判断なしに先へ進まない。
4. 実装計画を提示し、**確定を待つ【節目②】**。計画に含めること:
   ディレクトリ構成・採用ライブラリ・テスト方針・影響範囲(機能追加時)。
   新規開発のときは、ここで CLAUDE.md の TODO(Python バージョン・アプリの種類・
   フレームワーク・起動コマンド)を確定させ、CLAUDE.md を更新する。
5. 失敗するテストを先に書く(pytest)。その後に実装する。
   テストの実行は都度 `test-runner` エージェントに委譲し、失敗要約を受けて修正する。
6. `/verify` を実行する(整形・リント・テスト・起動確認)。
7. `git diff` で差分を取得し、`code-reviewer` エージェントに渡す:
   差分・レビュー観点(バグ/セキュリティ/エラー処理/テスト妥当性)・要件の3行要約。
   **実装時の会話コンテキストは渡さない。**
   「要修正」指摘があれば修正して手順6〜7を繰り返す(最大3周。解消しなければ
   停止して人間に報告する)。
8. テスト結果(通過数)とレビュー結果(要修正ゼロの証跡)を要約して
   完了報告する【節目③】。

## 注意

- 節目以外でユーザーに確認しない(「節目だけ確認」の要件)。
  逆に、競合調査を行った場合は節目①を、どの依頼でも節目②③を飛ばさない
  (手順1の基準で調査を省略した場合のみ、節目は②③の2つになる)。
- pytest の生ログをメイン会話に貼らない。必ず test-runner の要約を使う。
- テストを削除・スキップして通過扱いにしない(CLAUDE.md の禁止事項)。

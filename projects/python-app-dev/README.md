# python-app-dev — Python アプリ開発エージェント環境

要件を渡すと、競合(既存の類似製品・OSS・ライブラリ)を先に調査して報告し、
承認を得てから計画→テスト先行実装→検証→独立レビューまで進める Python 開発環境です。
人間の確認は3つの節目だけです。

## 最初の一歩

**`tools/request-form.html` をブラウザで開いてください。** 依頼種別(新規アプリ開発/
機能追加)を選んで項目を埋め、「指示文を生成」を押すと、そのまま Claude Code に
貼り付けられる指示文(要件定義書)ができます(.md 保存してパスを渡してもOK)。

フォームを使わず直接指示することもできます:

```
/develop TODOリストを管理できるWebアプリを作りたい。ユーザーは自分だけ。
```

## 仕事の流れと3つの節目

```
/develop → 競合調査 → 【節目① 作る/既存を使うの判断】
        → 実装計画 → 【節目② 計画の確定】
        → テスト先行実装 → /verify → 独立レビュー(要修正ゼロまで)
        → 【節目③ 完成報告】
```

節目以外は自律で進みます。テストが通り、code-reviewer の「要修正」がゼロになるまで
完了になりません。

## スキルとエージェント

| 構成物 | 役割 | モデル |
|---|---|---|
| `/develop` スキル | 新規開発・機能追加の一気通貫ワークフロー | (メイン) |
| `/verify` スキル | 整形・リント・テスト・起動確認 | (メイン) |
| `competitor-researcher` | 実装前の競合・ライブラリ Web 調査 | sonnet |
| `code-reviewer` | 読み取り専用の独立コードレビュー(品質ゲート) | opus |
| `test-runner` | pytest/ruff の実行と結果要約(生ログを会話に持ち込まない) | haiku |

モデルの割り振りはトークン効率のためです: 高頻度の定型作業(テスト実行)は haiku、
調査は sonnet、誤りコストが最大の最終レビューだけ opus を使います。

## 初回にやること

`CLAUDE.md` の技術スタック欄(Python バージョン・アプリの種類・フレームワーク・
起動コマンド)は TODO になっています。最初の `/develop` の計画確定時(節目②)に
エージェントが確定させて埋めます。先に決まっているなら手で埋めても構いません。

## 前提ツール

- Python + pytest + ruff(`pip install pytest ruff`)
- 保存のたびに `ruff format` が自動実行されます(.claude/settings.json のフック)。
  ruff 未導入でもエラーにはなりません。

## 既存リポジトリで使うには

```bash
cp -r .claude/ CLAUDE.md tools/ <あなたのリポジトリ>/
```
